About

Each language has conventions and styling guilds, this is a place to find good resources for the styling, naming, and other conventions for a language.

C
Convention Example
The "pointer" qualifier, '*', should be with the variable name rather than with the type. char *s, *t, *u;
Comments that describe data structures, algorithms, etc., should be in block comment form with the opening /*
Global variables should be prepended with a 'g_'. Logger g_log;
Structs are often nouns. By making function names verbs and following other naming conventions programs can be read more naturally.
Usually every function performs an action, so the name should make clear what it does: check_for_errors() instead of error_check(), dump_data_to_file() instead of data_file(). This will also make functions and data objects more distinguishable.
Don't use floating-point variables where discrete values are needed.
Courtesy of https://www.doc.ic.ac.uk/lab/cplus/cstyle.html and https://users.ece.cmu.edu/~eno/coding/CCodingStandard.html
Javascript
Convention Example
Always put spaces around operators ( = + - * / ) var x = y + z;
Always end a simple statement with a semicolon. var values = ["Volvo", "Saab", "Fiat"];
  • Put the opening bracket at the end of the first line.
  • Use one space before the opening bracket.
  • Put the closing bracket on a new line, without leading spaces.
  • Do not end a complex statement with a semicolon.
 For functions, loops, conditionals.
For readability, avoid lines longer than 80 characters.
Use camelCase for identifier names (variables and functions).
All names start with a letter.
Courtesy of w3schools, http://www.w3schools.com/js/js_conventions.asp
Java
Convention Example

When an expression will not fit on a single line, break it according to these general principles:

  • Break after a comma.
  • Break before an operator.
 someMethod(longExpression1, longExpression2, longExpression3, longExpression4, longExpression5);
Always end a simple statement with a semicolon. The // comment delimiter can comment out a complete line or only a partial line.
A return statement with a value should not use parentheses unless they make the return value more obvious in some way.
  • return;
  • return myDisk.size();
  • return (size ? size : defaultSize);
A block comment should be preceded by a blank line to set it apart from the rest of the code.
Put declarations only at the beginning of blocks.
Class names should be nouns, in mixed case with the first letter of each internal word capitalized. class ImageSprite;
Methods should be verbs, in mixed case with the first letter lowercase, with the first letter of each internal word capitalized. getBackground();
The names of variables declared class constants and of ANSI constants should be all uppercase with words separated by underscores ("_"). static final int MIN_WIDTH = 4;
Courtesy of oracle, http://www.oracle.com/technetwork/java/codeconvtoc-136057.html
http://www.oracle.com/technetwork/java/codeconventions-135099.html
Scheme
Convention Example
Predicate names end in a question mark ( ? ) eq?, zero?, and string=?. The common numeric comparators =, <, >, <=, and >= are exceptions to this rule.
The names of most character, string, and vector procedures start with the prefix char "-". string-append

Identifiers may be formed with

  • the lowercase letters a through z
  • the uppercase letters A through Z
  • the digits 0 through 9
  • the characters ? ! . + - * / < = > : $ % ^ & _ ~
 hi, Hello, n, x, x3, and ?$&*!!! are all identifiers
Keywords, variables, and symbols are collectively called identifiers.
Structured forms and list constants are enclosed within parentheses (a b c) or (* (- x 2) y)
The boolean values representing true and false are written as #t and #f. Scheme conditional expressions actually treat #f as false and all other objects as true.
Vectors are written similarly to lists, except that they are preceded by #( and terminated by ) #(this is a vector of symbols)
Strings are enclosed in double quotation marks
Characters are preceded by #\ #\a
Comments may appear on any line of a Scheme program, between a semicolon ( ; ) and the end of the line
Courtesy of scheme.com, http://www.scheme.com/tspl2d/intro.html
Ruby
Convention Example
Use two spaces per indentation level (aka soft tabs). No hard tabs.
Prefer string interpolation instead of string concatenation: email_with_name = "#{user.name} <#{user.email}>"
Never use then for multi-line if/unless
Don't use parentheses around the condition of an if/unless/while
Avoid return where not required.
Use spaces around the = operator when assigning default values to method parameters def some_method(arg1 = :default, arg2 = nil, arg3 = [])
Courtesy of Ruby Style guild, https://github.com/styleguide/ruby
Python
Convention Example
Block comments generally apply to some (or all) code that follows them, and are indented to the same level as that code. Each line of a block comment starts with a # and a single space (unless it is indented text inside the comment).
Imports should usually be on separate lines import os
import sys

Blank lines

  • Surround top-level function and class definitions with two blank lines.
  • Method definitions inside a class are surrounded by a single blank line.
  • Use blank lines in functions, sparingly, to indicate logical sections.
Limit all lines to a maximum of 79 characters.
Always surround binary operators with a single space on either side: assignment ( = )
While sometimes it's okay to put an if/for/while with a small body on the same line, never do this for multi-clause statements.
Courtesy of Python.org, https://www.python.org/dev/peps/pep-0008/
